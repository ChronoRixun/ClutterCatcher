# M6.1 — HEIC + Photo Cache GC (sub-plan)

Date: 2026-07-19 · Status: locked for implementation · Follow-up slice of M6
item photos (see `planning/m6-photos-plan.md`, P1–P14, and OPEN_ITEMS
DL39–DL50). Both features resolved with Owen 2026-07-19 (Questions for Owen,
Run 4 — items 1 and 3).

## 1. Scope

Two small, local-only changes:

- **A. HEIC encoding** in `PhotoStore` (supersedes the JPEG interim, DL42).
- **B. "Clean Up Unused Photos"** — an on-demand GC sweep in Settings that
  deletes cached photo files no longer referenced by anything (closes the
  DL44 bounded cache leak).

**No sync contract change. No schema change. No migration. No CloudKit
deploy.** Asset bytes are opaque to CloudKit and to peers; `photo_asset_ref`
semantics (P6) are untouched.

## 2. Decisions (continue the P-series)

- **P15 — HEIC via ImageIO with quality control; JPEG encode fallback.**
  Encode with `CGImageDestination` (`UTType.heic`,
  `kCGImageDestinationLossyCompressionQuality` = 0.8). Do NOT use
  `UIImage.heicData()` (no quality parameter). If HEIC encoding fails or is
  unavailable (paranoia only — the whole fleet is iOS 26 hardware), fall back
  to the existing `jpegData(compressionQuality:)` path. This finally matches
  P12's original "HEIC preferred / JPEG fallback" wording.
- **P16 — File naming stays `Photos/<ref>.jpg` / `<ref>_thumb.jpg`,
  regardless of actual encoding.** The extension is a cache-internal detail:
  decode is content-based (`UIImage(data:)` / `UIImage(contentsOfFile:)`
  sniff bytes, not names), and inbound peer assets are already opaque bytes of
  unknown encoding copied verbatim (`ensureLocalFile`) — so naming by content
  would be a lie half the time anyway. Keeping the DL42 id→path contract
  byte-for-byte means zero lookup changes, zero migration, and mixed
  JPEG/HEIC files coexist silently. Document the semantics on `fileURL(for:)`:
  ".jpg here means 'image file'".
- **P17 — Thumbnails encode HEIC too** (same call, same fallback). Local-only,
  regenerable, trivial.
- **P18 — GC "referenced" set = items-table refs ∪ orphan-buffer refs.**
  A file pair (`<ref>.jpg` + `<ref>_thumb.jpg`) is live iff `ref` appears in
  (a) any non-nil `items.photo_asset_ref`, OR (b) the `photo_asset_ref` of any
  Item row serialized inside `orphaned_records` — P9 materializes orphan photo
  bytes at buffer time, and drain expects them on disk; a sweep that only
  checks the items table would eat bytes the drain needs. Everything else in
  `Photos/` whose name parses as `<ref>[_thumb].jpg` and isn't live → delete.
  (Container covers resolve through an item id, never a ref directly — the
  items-table set already covers them.)
- **P19 — Staged-capture safety: on-demand only + a 1-hour age guard.** The
  sweep runs solely from the Settings button (editor sheets cover the UI, so
  Settings and an open editor can't coexist today — but don't rely on that
  topology forever). Belt-and-suspenders: skip any file whose modification
  date is within the last hour. `sessionRefs` staged in an editor are
  DB-invisible until Save (DL43); the age guard makes the sweep safe even if
  navigation changes later. Never run on a timer or at background.
- **P20 — Settings UI.** A "Clean Up Unused Photos" button in the same
  section as "Re-download Photos" (or an adjacent one), disabled while
  running, reporting a transient result on completion: "Removed N files,
  freed X MB" (or "Nothing to clean up"). Failures per-file are logged and
  skipped, never surfaced as errors — a leftover file is the status quo, not
  a fault.

## 3. Design notes

- `PhotoStore` gains one internal `encode(_ image: UIImage) -> Data` seam used
  by both `importImage` and `writeThumbnail`; HEIC-with-JPEG-fallback lives
  there once.
- `PhotoStore` gains a pure sweep API, e.g.
  `sweepUnusedPhotos(keeping live: Set<String>, olderThan: Date) -> (files: Int, bytes: Int64)`
  — filesystem work only; it does not know where the ref set comes from.
- The caller (Settings, via a small helper on the database side) assembles the
  live set: one read of `SELECT DISTINCT photo_asset_ref FROM items WHERE
  photo_asset_ref IS NOT NULL`, plus decoding `orphaned_records` rows with the
  existing `OrphanedRecord` machinery and collecting item refs. Read-only DB
  access — the two write paths (DL20) are untouched by this entire slice.
- Existing full-size JPEGs on devices stay as they are (they only re-encode if
  the user replaces the photo, which mints a new ref anyway). Already-uploaded
  JPEG CKAssets stay JPEG. Nothing re-uploads.

## 4. Tests (Swift Testing)

- Encode: `importImage` still writes both files, downscales ≤2048, mints a
  fresh ref; written data decodes via `UIImage(data:)`. If asserting the HEIC
  container, check the ISO-BMFF `ftyp` brand bytes rather than the extension —
  but a decode round-trip is the required floor (encoder availability on CI
  simulators can vary; the fallback path must also be exercised by injection
  or a forced-failure seam if cheap, else document why not).
- Existing `PhotoStoreTests` keep passing unchanged (paths/extensions are
  P16-stable).
- Sweep: given referenced + unreferenced + orphan-buffered + fresh (<1h) file
  pairs → deletes exactly the unreferenced-and-old pair (full + thumb), counts
  and byte totals accurate; empty directory and no-`Photos/`-dir cases are
  no-ops; a second sweep is a no-op.
- Live-set assembly: items-table refs and orphan-buffered item refs both
  included; non-item orphans ignored.
