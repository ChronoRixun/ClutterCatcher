## Scope

### 1. Project configuration

`project.yml`: `TARGETED_DEVICE_FAMILY: "1,2"` (replace the stale
"iPhone-only for M0–M6" comment — this IS the M6 iPad work); iPad
supported orientations (all four, `~ipad` key); do **not** set

### 2. Layout adaptation — size classes, not device idiom

The standing rule from the plan: **size-class-driven adaptation, not a
stretched phone layout** — and equally, not a navigation rewrite. The
recommended shape, deviations logged as DLs:

- Keep the TabView; adopt `.tabViewStyle(.sidebarAdaptable)` so iPad
  gets the native sidebar/top treatment while iPhone is untouched.
- Forms and settings screens: readable-content width in regular width
  (no 12-inch-wide text fields).
- Rooms home: consider an adaptive grid of room tiles in regular width
  (the T11 accent-cycle tiles were made for it); the list stays on
  compact. Judgment call — a well-spaced list is acceptable, a
  full-width stretched list is not.
- Container detail, item editor, search: readable width + sensible
  column behavior; editors present as form sheets on iPad (verify
  sizing; mind DL59's presentation-timing lessons on the new idiom).
- Themed backgrounds (`themedScreen()`) must fill edge to edge at every
  size including Stage Manager's arbitrary widths; `themedRow()`
  surfaces must not assume compact metrics. Fen scales but caps (an
  88 pt fox, not a 400 pt one). Spot-check all six themes at iPad
  sizes — Classic must remain structurally untouched (DL54's no-op
  contract holds on iPad too).
- Popover anchors: on iPad, share sheets and confirmation dialogs
  present as popovers — audit every `confirmationDialog` and any
  share/print path (label PDF flow especially) for sane anchoring.
- Scanner: `DataScannerViewController` runs on iPad; verify the
  viewfinder + overlay card layout in landscape and Split View; the
  manual fallback is unchanged.
- Label sheet: the PDF preview finally has room — make sure it uses it
  (larger preview in regular width), print flow unchanged.
